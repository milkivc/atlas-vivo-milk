# Contributing to Atlas Vivo - Datasets

Thank you for your interest in contributing to the Atlas Vivo project. Your contributions help preserve Portugal's intangible cultural heritage and support open data initiatives.

---

## Code of Conduct

By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md). Please read it before contributing.

---

## How Can I Contribute?

### Reporting Bugs

If you find a bug, please open an issue on our [GitHub Issues](https://github.com/milkivc/atlas-datasets/issues) page with:
- A clear and descriptive title
- Steps to reproduce the issue
- Expected vs. actual behavior
- Screenshots if applicable
- Your environment (OS, browser, etc.)

### Suggesting Enhancements

For feature requests or enhancements:
- Open an issue with the "enhancement" label
- Provide a clear description of the proposed feature
- Explain why this feature would be valuable
- Include any relevant examples or mockups

### Contributing Code

1. Fork the repository on GitHub
2. Clone your fork locally:
   ```bash
   git clone https://github.com/your-username/atlas-datasets.git
   cd atlas-datasets
   ```
3. Create a branch for your changes:
   ```bash
   git checkout -b feature/your-feature-name
   ```
4. Make your changes and ensure they follow our guidelines
5. Commit your changes with a descriptive message:
   ```bash
   git commit -m "feat: add new feature"
   ```
6. Push to your fork:
   ```bash
   git push origin feature/your-feature-name
   ```
7. Open a Pull Request to the main repository

### Contributing Data

For contributing datasets:
- Ensure all data is properly licensed
- Include metadata in DataCite 4.4 format
- Anonymize any personal data
- Verify compliance with RGPD and AI Act
- Include provenance information

## Pull Request Guidelines

### Before Submitting

- Ensure your code follows our style guide
- All tests pass
- Code is properly commented
- Documentation is updated if needed
- You have signed your commits (GPG)

### Pull Request Template

Your PR should include:
- A clear title describing the change
- A description of what was changed and why
- Reference to any related issues
- Screenshots if UI changes are involved

---

## Style Guides

### Git Commit Messages

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally

### Code Style

- Follow the existing code style
- Use consistent indentation (2 spaces for JSON/YAML, 4 spaces for Python)
- Keep lines under 120 characters when possible
- Use descriptive variable and function names
- Comment complex logic

### Documentation Style

- Use Markdown for all documentation
- Keep documentation up to date
- Use clear and concise language
- Include examples where helpful

---

## Documentation

All contributions that change functionality or add new features should include updates to:
- README.md (if applicable)
- Relevant documentation files
- Code comments

---

## Legal Compliance

All contributions must:
- Comply with RGPD (General Data Protection Regulation)
- Respect copyright and licensing terms
- Not include personal data without proper anonymization
- Follow the AI Act requirements for high-risk systems

### Data Protection

- Never commit personal data to the repository
- Anonymize all sensitive information
- Use placeholder values for examples
- Follow the principle of data minimization

### Licensing

- All code contributions are licensed under EUPL-1.2
- All data contributions should specify their license
- Respect third-party licenses and copyrights

---

## Security

If you discover a security vulnerability, do not open a public issue. Instead, please send an email to security@associacaomilk.pt with details of the vulnerability.
